해당 페이지의 기능 상세 디스크립션을 작성하여 `.desc/[page-name].md` 파일로 저장한다.
피그마에서 개발자/디자이너에게 전달할 용도로, 화면 상태별 번호 핀 + 기능 설명 형식으로 작성한다.

## Steps

1. **브레인 읽기** : `.brain/` 디렉토리에서 해당 페이지의 brain 파일을 읽는다.
   - 컴포넌트 구조, 사용자 흐름, 엣지 케이스, 권한 규칙을 파악한다

2. **공통 규칙 읽기** : `.brain/cross-page-rules.md` 를 읽는다.
   - 역할 체계, 목업 데이터 기준값, ENUM 값 등 페이지 간 일관성 확인

3. **스키마 읽기** : `.brain/db-schema.md` 를 읽는다.
   - 해당 페이지에서 사용하는 테이블과 필드를 파악한다

4. **HTML 파일 읽기** : 해당 HTML 파일을 읽어 실제 구현된 기능을 파악한다.
   - brain 스펙과 실제 구현의 차이가 있으면 실제 구현 기준으로 작성한다

5. **디스크립션 작성** : 아래 형식을 따른다.

## 작성 형식

### 화면 상태 분리
- 역할별 뷰가 있으면 각 역할별로 화면을 나눈다 (예: 원장 뷰, 교사 뷰, 교사-반없음 뷰)
- 모달이 있으면 별도 화면으로 분리한다
- 특수 상태 (빈 상태, 필터 적용 등)도 별도 화면으로 분리한다

### 각 화면 내 구성
- `## 화면 N: [페이지명] — [상태 설명]` 형식의 제목
- 기능 영역별 `**① 영역명**` 형식의 번호 핀
- 각 핀 아래에 기능 상세 설명:
  - 해당 영역의 UI 구성 요소
  - 사용자 인터랙션 (클릭, 입력, 선택 등)
  - 데이터 표시 규칙 (필터, 정렬, 포맷)
  - 조건부 노출 규칙 (권한, 상태에 따른 차이)
  - 엣지 케이스 처리
  - 목업 데이터 예시 (테이블 형태로)

### 부록 섹션
- **권한 매트릭스**: 역할별 기능 접근 권한 요약 테이블
- DB 스키마, 테이블명, 쿼리 힌트 등 기술적 구현 세부사항은 **절대 포함하지 않는다**. 이 문서는 기능 동작을 전달하는 용도이며, DB 설계는 개발자의 영역이다.

## 작성 원칙

- 디자인 스펙(색상, 폰트, 간격)이 아닌 **기능 동작**에 집중한다
- 색상/스타일은 조건부 표시의 구분이 필요할 때만 언급한다 (예: "미열람=빨간 태그, 열람=초록 태그")
- 프로토타입 전용 요소(역할 전환 바)는 실서비스에서 제거됨을 명시한다
- 목업 데이터는 `.brain/cross-page-rules.md`의 기준값과 일치해야 한다
- 개발자가 이 문서만 보고 기능을 구현할 수 있을 정도로 상세하게 작성한다

## 출력

- 파일 경로: `.desc/[page-name].md`
- 예: dashboard → `.desc/dashboard.md`

## Usage

User says: `/figma-desc [page-name]`

Example: `/figma-desc dashboard` → reads brain + HTML → saves `.desc/dashboard.md`

If the page name is ambiguous, list matching brain files and ask which one to use.
If no page name is provided, list all available pages and ask which one to use.
