페이지의 기능을 역할별 시나리오 단위로 테스트하고 결과 리포트를 출력한다.

`review`가 스펙 vs 구현의 정적 비교라면, `test`는 **실제 사용자 흐름을 시뮬레이션**하는 동적 검증이다.

---

## Steps

### 1. 준비

- 대상 HTML 파일을 읽는다.
- `PERMISSION-MATRIX.md`를 읽어 해당 페이지의 역할별 권한을 파악한다.
- 해당 페이지의 `.brain/*.md` 파일을 읽어 사용자 흐름과 엣지 케이스를 파악한다.

### 2. 테스트 케이스 구성

아래 4개 카테고리로 테스트 케이스를 자동 생성한다.

#### A. 역할별 권한 테스트 (Permission Tests)
각 역할(`super`, `org_admin`, `teacher`)에 대해:
- 접근 허용/차단이 코드에 구현되어 있는가?
- 수정/삭제 버튼이 권한 없는 역할에서 숨겨지거나 비활성화되는가?
- 조회 전용 배지·안내 문구가 올바른 역할에서만 표시되는가?
- 담당 반 범위 제한이 적용되는가? (`assignedClasses` 세션 체크)

#### B. UI 상태 테스트 (State Tests)
- 빈 상태(Empty State): 데이터가 없을 때 안내 메시지가 있는가?
- 로딩 상태: 비동기 처리를 가정한 스켈레톤 또는 스피너가 있는가? (프로토타입에서는 생략 가능 — 명시)
- 에러 상태: 잘못된 입력, 필수 항목 누락 시 피드백이 있는가?
- 모달/패널 열기·닫기: ESC, 배경 클릭, 닫기 버튼 3가지 모두 구현되었는가?

#### C. 네비게이션 흐름 테스트 (Navigation Tests)
- 사이드바 활성 항목이 현재 페이지와 일치하는가?
- 페이지 내 링크/버튼이 올바른 `href`로 연결되어 있는가?
- sessionStorage pre-fill 키를 설정하고 대상 페이지로 이동하는 흐름이 구현되어 있는가?
  - `noticePrefill` → `operation-notice-board.html`
  - `medicinePrefill` → `operation-medicine.html`
- 브레인에 명시된 다른 교차 페이지 이동 흐름이 있으면 동일하게 검증

#### D. 데이터 일관성 테스트 (Data Tests)
- 목업 데이터의 필드 구조가 `.brain/db-schema.md`의 테이블 정의와 일치하는가?
- 역할 전환 시 동일한 데이터 소스를 재사용하는가? (역할마다 별도 하드코딩 금지)
- 목업 ID 형식이 일관적인가? (예: CHD001, CLS001)

### 3. 테스트 실행 및 결과 판정

각 테스트 케이스를 코드에서 직접 확인하고 아래 기준으로 판정한다:

| 결과 | 기준 |
|------|------|
| ✅ PASS | 코드에 명확히 구현됨 |
| ⚠️ PARTIAL | 일부만 구현되었거나 조건 분기가 불완전함 |
| ❌ FAIL | 구현 없음 또는 반대로 동작 |
| 🔲 SKIP | 프로토타입 범위 초과 (예: 로딩 상태) — 사유 명시 |

### 4. 리포트 출력

아래 형식으로 리포트를 출력한다:

```
## 🧪 기능 테스트 리포트 — [페이지명]
테스트 일시: YYYY-MM-DD  |  파일: [파일명]

### 요약
✅ PASS: N  ⚠️ PARTIAL: N  ❌ FAIL: N  🔲 SKIP: N

### A. 역할별 권한 테스트
| # | 테스트 케이스 | super | org_admin | teacher | 결과 |
|---|-------------|:-----:|:---------:|:-------:|:----:|
...

### B. UI 상태 테스트
...

### C. 네비게이션 흐름 테스트
...

### D. 데이터 일관성 테스트
...

### 수정 필요 항목
> FAIL·PARTIAL 항목만 모아서 간략히 설명
```

### 5. 수정 제안

리포트 출력 후 사용자에게 묻는다:
> "FAIL/PARTIAL 항목을 수정할까요? 항목 번호를 지정하거나 '전체' 라고 말씀해주세요."

사용자가 수정을 요청하면 해당 항목만 집중적으로 수정한다.
`review` 스킬과 달리 전체 재빌드는 하지 않는다 — **핀포인트 수정**만 한다.

## Usage

```
/project:test [page-name]
```

**예시**
- `/project:test dashboard` → `operation-dashboard.html` 테스트
- `/project:test invitation` → `operation-invitation.html` 테스트
- `/project:test all` → 구현된 모든 `operation-*.html` 순차 테스트 (요약 리포트)

**인수 없이 실행 시** → 구현된 페이지 목록을 보여주고 선택 요청

## 프로토타입 테스트 원칙

이 프로젝트는 Vanilla HTML/JS 프로토타입이므로 아래 항목은 테스트 범위에서 제외한다:
- 실제 API 호출 / 네트워크 에러 처리
- 서버 사이드 권한 검증
- 브라우저 간 호환성 (Chrome 기준으로만 검증)
- 성능 측정 (LCP, FCP 등)

위 항목이 없는 경우 `🔲 SKIP — 프로토타입 범위 초과`로 표기한다.
